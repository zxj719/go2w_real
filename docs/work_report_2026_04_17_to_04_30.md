## 2. 阶段总体摘要

本阶段的核心目标，不是单独调通一套导航栈，而是配合“大脑”系统，在 GO2W 上完善“小脑”能力：把大脑下发的讲解任务、点位目标、暂停/中断和安全控制，转成机器人端稳定可执行的移动、避障、到点、状态回传和紧急停止能力。最终业务目标是让 GO2W 搭载 Thor 后，在公司展厅中配合语音讲解，帮助来宾完成展厅介绍。

### 2.1 主线任务一：配合大脑完善 GO2W 小脑

本阶段围绕“大脑负责意图和讲解，小脑负责运动执行和安全闭环”的分工推进。GO2W 侧重点包括：

- 对外接口：WebSocket 导航执行器、`navigate_to`/`abort_navigation` 协议、进度/到达/错误回传、POI/Waypoint 解析。
- 本地执行：Nav2 goal 派发、near-goal 判定、目标抢占、中断后重新下发目标、headless 启动和 readiness 检查。
- 安全控制：Operator Console、手机端 E-STOP、持续 `/cmd_vel=0`、停止当前导航子进程和全部运行 profile。
- 现场排障：debug bag、TF/Scan/XT16/Nav2 readiness、运行态参数检查、RViz/Foxglove 可视化入口。

从联调链路看，目标流程是：

`来宾语音/讲解任务 -> 大脑语义与讲解编排 -> 目标点/控制指令 -> GO2W 小脑导航执行 -> 到点/失败/进度状态回传 -> 大脑继续讲解或重新规划`

### 2.2 主线任务二：与大脑语音控制工程师联调

与负责大脑语音控制的工程师联调时，GO2W 侧主要完成了以下接口和调试支撑：

- 明确机器人端 `navigation_executor.py` 是 WebSocket client，由机器人主动连接大脑/调度端 server。
- 固化 `target_id`/POI 到 `config/go2w_waypoints.yaml` 的解析方式，保证语音意图、页面点位和实际导航目标一致。
- 支持 `abort_navigation` 和新 `navigate_to` 抢占旧目标，使语音中断、换点、重新讲解具备机器人端执行基础。
- 回传 `on_progress`、`on_arrived`、`on_error`，让大脑侧可以基于机器人状态继续讲解、重试或提示人工接管。
- 用手机端 Operator Console 和 debug bag 作为联调兜底工具：当语音链路异常时，可直接本地启动导航栈、发送 POI、录包并查看可视化。

### 2.3 关键卡点：Thor 负载遮挡 LiDAR 后向视场

阶段内最主要的现场卡点来自 Thor 负载对 LiDAR 视场的影响。机器人不背负 Thor 时，公司展厅完整路线当前测试未出现一次失败；背负 Thor 后，LiDAR 后半部分被遮挡，基于纯点云匹配的 odom 在长走廊中更容易漂移，进而触发 slam_toolbox/Nav2 的重定位 recovery。当前背负 Thor 跑完整个展厅的成功率约为 `60%` 到 `70%`。

该问题的本质不是单个参数过大或过小，而是传感器几何和场景退化叠加：

- Thor 遮挡导致后向点云可用结构减少，scan matching 约束变弱。
- 公司展厅长走廊特征重复，纯点云匹配在走廊方向容易出现累计漂移。
- 漂移积累后，`map -> odom` 修正或重定位 recovery 会引入突然转向、丢定位或路径执行中断。
- 仅靠调 Nav2 controller 或 goal checker，无法从根上解决“观测不足 + 长走廊退化”的问题。

### 2.4 IMU/机身里程计补偿尝试及结论

为解决纯点云 odom 在 Thor 场景下的漂移问题，本阶段测试了机器人本体和 LiDAR 侧 IMU/状态源，但结果不能作为直接替代方案。结论是：这些数据源可以继续作为 shadow 对照或低权重辅助，但还不能简单高权重融合后直接解决 Thor 背负下的长走廊漂移。

| 数据源 | 关键测试结果 | 阶段结论 |
| --- | --- | --- |
| `/lowstate.imu_state` | 频率约 `500 Hz`；`gyro.z` 静止偏置稳定在约 `-0.009015 rad/s`，折算约 `-31.0 deg/min`；加速度模长约 `9.44 m/s^2` | 姿态输出有参考价值，但 yaw rate 必须扣偏置或给较大 covariance，不能直接高信任融合 |
| `/utlidar/imu` | 频率约 `249.8 Hz`；静止 `gyro.z mean` 约 `0.000610 rad/s`，yaw drift 约 `-0.0022 deg/min`；但 z 轴方向与 base yaw 相反，covariance 全 0 | 静止质量好于 `/lowstate`，但接入前必须做轴向修正、外参确认和 covariance 补齐 |
| `/lf/sportmodestate` | 静止 20 秒 position xy 漂移约 `0.1534 m`；追加 60 秒静止复测漂移约 `0.4469 m/min` | position/velocity 不能当高可信 odom 真值，不适合直接替代 RF2O 或作为闭环距离依据 |

因此，IMU/里程计方向的阶段判断是：保留 RF2O/点云 odom 作为当前主链路，同时推进 `/utlidar_imu_base`、`/lowstate_imu`、`/sport_odom` 的低权重 EKF shadow 对比；在轴向、外参、偏置、covariance 未完成验证前，不把本体 IMU 作为生产级修复手段。

### 2.5 阶段工程成果

阶段成果可以概括为七条主线：

1. 建立并完善 GO2W 实机 Nav2 + SLAM + RF2O 的统一启动链路。
2. 新增 `motion_executor` 分段运动执行器，并采用 `shadow -> active-minimal` 的低风险接管路线。
3. 打通 WebSocket 导航执行器、Waypoint/POI 导航、手机端 Operator Console 三类指令入口，支撑大脑语音控制联调。
4. 建立 headless 部署入口，收敛配置来源，并补齐 XT16、TF、Scan、Nav2 readiness 等启动前检查。
5. 围绕 Thor 背负后的长走廊漂移问题，补齐 debug bag、重定位参数、地图/profile 和运行态检查工具。
6. 对 `/lowstate`、`/lf/sportmodestate`、`/utlidar/imu` 等机身状态源做实机测试，形成 IMU/里程计融合的谨慎方案。
7. 补齐调试 bag、运行态参数检查、单元测试和现场报告文档，降低后续排障和交接成本。

## 7. 当前结论

1. 本阶段主线是配合大脑系统完善 GO2W 小脑能力，使机器人能承接语音讲解系统下发的展厅导航任务。
2. GO2W 实机导航已从多终端手动操作，收敛为可配置、可等待 readiness、可录包、可部署的 headless 入口，并能通过 WebSocket/手机端 Operator Console 派发 POI。
3. Nav2 + RF2O + slam_toolbox 仍是当前主线，其中 localization 方向优先走无 AMCL 的 slam_toolbox 路线。
4. 不背负 Thor 时，公司展厅完整路线当前测试未出现失败；背负 Thor 后，由于 LiDAR 后向视场被遮挡，长走廊中纯点云 odom 容易漂移并触发重定位 recovery，完整路线成功率约 `60%` 到 `70%`。
5. `/lowstate.imu_state`、`/lf/sportmodestate`、`/utlidar/imu` 已完成初步实机测试；结果说明机身 IMU/里程计不能直接作为高可信生产 odom，但可继续做低权重辅助或 shadow 对照。
6. `motion_executor` 已建立模块基础，但正确策略仍是先 shadow 观察，再逐步 active 接管，不建议一次性启用完整复杂语义。
7. 手机端 Operator Console 已能支撑本地局域网 POI 派发、headless 启动、debug bag 调参和强制紧急停止，是现场联调和大脑链路异常时的重要兜底入口。
8. 后续排障必须继续坚持“源码配置、launch 临时参数、运行态 ROS 参数、action feedback、debug bag”同时检查。

## 8. 遗留问题与风险

- Thor 背负后遮挡 LiDAR 后半部分，是当前展厅导览成功率从“不背负时未见失败”下降到约 `60%` 到 `70%` 的主要风险点。
- 长走廊场景对纯点云匹配不友好，RF2O/scan matching odom 在走廊方向容易累计漂移，单纯调 controller 参数不能从根上解决。
- 重定位 recovery 仍可能在 odom 漂移后触发突然转向、丢定位或路径执行中断，需要结合传感器安装、融合和策略层共同优化。
- 大脑语音控制联调仍需要继续验证异常链路：语音中断、重复派点、导航失败重试、人工接管和讲解流程恢复。
- `motion_executor active` 仍需更多实机安全验证，尤其是 recovery、final yaw、复杂路径下的状态切换。
- `/utlidar/imu` 到 `base` 的完整外参仍需正式标定，目前只确认了 yaw rate z 方向需要修正。
- `/lowstate.imu_state` 的 gyro.z 静止偏置稳定，但动态条件下的偏置和 covariance 还需进一步验证。
- `/lf/sportmodestate` position/velocity 静止漂移明显，不能直接作为闭环复位或里程计真值。
- 无 AMCL 的 slam_toolbox localization 仍需要在动态障碍物更多、长时间运行和多地图切换下继续验证。
- `test` 分支中的 4 月 30 日补充工作尚未出现在当前 `master`，合入前需要重新跑相关测试并检查冲突。
- Operator Console 当前主要面向局域网使用，HTTP API 无鉴权，部署时应确保网络隔离。
